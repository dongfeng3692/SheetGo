

     一、Agent 框架调研
    
     1. pandas-ai（v3.0.0-alpha）— 数据分析专用 Agent
    
     设计思路: "SQL-first + LLM 代码生成 + 安全执行"
     - 本地数据通过 DuckDB 注册为虚拟表，统一走 SQL 路径
     - LLM 生成 SQL + Python 后处理代码，而非直接 pandas 操作
     - 已停止维护（最后实质提交 2025.10），v3.0.0 从未发布稳定版
    
     核心流程:
     用户提问 → DataFrame 序列化为 <table> XML（schema + 前5行）
              → Jinja2 Prompt → LLM 生成代码（调用 execute_sql_query()）
              → AST 代码清洗 → exec() 执行 → 解析 result dict
              → 失败重试（最多3次，错误反馈 LLM 修正）
    
     关键设计: SQL 注入防护（正则+AST双重）、Docker 沙箱、语义层 Schema（28种转换）、有界对话记忆（10条）。
    
     2. claw-code — Claude Code 的 Rust 重实现
    
     设计思路: Trait 驱动的类型安全 Agent 框架。ConversationRuntime<C, T> 泛型参数化 LLM 和工具。9 个 Rust crate，Hook
      三阶段（Pre/Post/Failure），五级权限引擎，会话压缩。
    
     3. opencode-dev — 开源 AI 编程 Agent
    
     设计思路: Effect-TS 函数式 + 事件溯源 + Client/Server 架构。多 Agent 类型（build/plan/explore）、25+ LLM
     提供商、Skill 系统（Markdown 文件）、操作快照/回滚。

     对本方案的关键启发
    
     启发 1: minimax 的 XML 保留路线（解决格式丢失问题）
    
     核心洞察: openpyxl round-trip 会丢失 VBA、PivotTable、Sparkline、Chart 格式等。minimax 的"禁止 openpyxl
     写入"路线虽然精确但太底层。
    
     本方案采用混合策略:
     - 读取: 用 calamine（Rust 实现，极快）+ openpyxl（格式信息）
     - 结构化修改（增删行列、写公式）: 用 minimax 式的 XML 辅助脚本（shift_rows、add_column、insert_row）
     - 新建文件: 用 minimax 的 XML 模板系统（预定义样式 slot）
     - 纯数据查询/分析: 用 DuckDB SQL（不走 openpyxl）
    
     启发 2: kimi 的 Per-Sheet 验证循环
    
     核心洞察: 不要等所有 sheet 做完再验证，而是每个 sheet 做完立即 recheck +
     reference-check，零错误后才做下一个。避免错误级联。
    
     本方案采用: 上传文件后预加载阶段就运行全量验证（公式错误、引用错误、结构完整性），结果缓存供后续操作使用。每次修
     改后自动增量验证。
    
     启发 3: 各家 Skill 的工具化思路
    
     核心洞察: 4 家 skill 都预置了 Python/编译辅助脚本（recalc、validate、inspect、pivot），而不是让 LLM 从零写代码。
    
     本方案: 将所有 Excel 操作封装为预置工具函数（而非让 LLM 写 openpyxl 代码），LLM 只做决策和调用。这比 pandas-ai
     的"LLM 写代码再执行"更安全可控。

     启发 5: 禁用函数列表（兼容性保障）
    
     Excel 2019 及更早版本不支持: FILTER、UNIQUE、XLOOKUP、SORT、SEQUENCE、LET、LAMBDA、XMATCH。需要用
     INDEX/MATCH、SUMIF/COUNTIF 等替代方案。kimi 和 openai 两家都明确列出了替代方案
